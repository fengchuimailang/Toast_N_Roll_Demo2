# 🔎 Review Roadmap - 当前问题与修复优先级

> 创建日期: 2026-03-27  
> 状态: ✅ 已确认  
> 目标: 将当前版本的核心问题沉淀为可执行的整改路线图

---

## 目的

这份文档用于回答一个直接问题：

> 当前这个游戏原型，最值得优先解决的问题是什么？

结论不是“继续堆内容”，而是先把已有机制压成一个真正可玩的闭环。

当前版本已经具备：
- 上浮补位
- 六阶食材进阶
- 多口味合成
- 顾客系统
- 道具系统
- 50关配置

但这些机制之间还没有完全形成统一的玩家目标、胜负判断和难度递进。

---

## 核心判断

当前项目的主要问题可以概括为一句话：

**机制已经很多，但主目标、关卡差异和玩家反馈还没有压成一条清晰主线。**

这会直接带来三个结果：
- 玩家不知道这局到底为什么赢、为什么输
- 关卡配置很多，但实际体验差异不足
- 中后期复杂度上升很快，但反馈层还不够清晰

---

## P0

### 1. 关卡结束条件已经切到顾客池闭环

当前版本已采用：
- 固定顾客池
- 回合制耐心消耗
- 所有顾客结算完毕后结束关卡
- 无步可走且仍有未结算顾客时失败

这条主线已经比旧版更清晰，但后续仍需继续验证：
- 顾客总数和耐心曲线是否合理
- 服务率与分数的映射是否足够直观
- 后期顾客离场是否过快

建议：
- 保持顶部展示剩余待结算顾客数量
- 在结算页展示 `served / missed / total`
- 后续补顾客系统回放测试

### 2. 失败条件定义不统一

问题：
- 当前失败条件已统一为“无可移动步骤且仍有未结算顾客”
- 但还需要继续确认是否保留 `moveLimit` 作为附加限制

风险：
- 设计稿和实际玩法不一致
- 难度曲线无法准确调试
- 中后期关卡无法稳定控制挑战性

建议：
- 保持“死盘失败”作为主失败条件
- 如需增加难度，再单独引入 `moveLimit`
- 不再恢复秒级倒计时模型

### 3. 顾客系统与礼盒系统已完成职责拆分

当前版本：
- 礼盒是服务顾客的手段
- 顾客池是关卡结算的主线
- 金币与分数主要由服务成功顾客产生

后续关注点：
- 不同口味礼盒的收益差是否足够明显
- 三口味阶段是否需要更强反馈来解释“做哪位顾客更赚”

---

## P1

### 1. 关卡参数还没有全部转化为实际体验差异

问题：
- 新字段已经切换为 `totalCount / basePatience / queuePatienceOffset / entryGraceTurns / demandCountRange / allowedFlavors`
- 但这些字段的数值曲线还没有经过系统调优

风险：
- 关卡看起来很多，玩起来像同一种关卡
- 后续策划调数值没有抓手
- QA 无法判断关卡是否真按配置运行

建议：
- 先让顾客系统完全配置化
- 再把评分、三星线、生成节奏接进结算流程

### 2. 三口味与老虎机阶段的认知压力偏高

问题：
- 第21关后引入第三口味
- 同时又叠加老虎机口味决策
- 对玩家来说，复杂度增长远高于普通“三消加一种新元素”

风险：
- 玩家不理解“为什么这次合成出这个口味”
- 系统会被误解为随机而不是规则驱动
- 中期留存会受影响

建议：
- 把老虎机触发条件做成明确提示
- 在合成前后加入更强的视觉反馈
- 单独做1到2个低压力教学关，专门解释三口味规则

### 3. 当前安全生成策略可能压低爽感

问题：
- 开局避免直接三连是合理的
- 但当前还会在一定范围内主动规避“附近同类”
- 这可能导致盘面过散

风险：
- 玩家更难规划中短链
- 上浮补位虽然成立，但连锁爽感不足
- 棋盘会显得“稳定但不兴奋”

建议：
- 保留基础安全生成
- 降低对附近同类的限制强度
- 用可回放测试验证“无脏盘”和“可形成连锁机会”的平衡点

---

## P2

### 1. 测试护栏还没有覆盖玩法闭环

当前 Harness V1 已覆盖：
- 关卡配置合法性
- 上浮补位逻辑

还未覆盖：
- 顾客系统回放
- 老虎机决策回放
- 通关/失败判定
- 步数与时间限制

建议：
- 将上述内容纳入 Harness V2
- 先补规则性测试，再补 UI 端到端验证

### 2. UI 反馈层还不够承载当前复杂度

问题：
- 当前系统已经不只是简单三消
- 但 UI 还没有把目标、限制、收益、特殊规则做出足够清晰的层级

建议：
- 顶部优先展示本关目标
- 明确剩余步数或时间
- 明确顾客是否影响通关
- 明确老虎机触发规则和结果

### 3. 中后期难度曲线需要二次审视

问题：
- 7x7
- 三口味
- 多顾客
- 更紧的步数限制

这些维度如果叠加过快，会让难度从“有挑战”直接跳成“看不懂”

建议：
- 把第21关后的复杂度拆开释放
- 避免在同一小段区间同时提升3个以上压力源

---

## 推荐执行顺序

建议严格按下面顺序推进：

1. 先做 `通关/失败/目标显示` 闭环
2. 再做 `顾客系统配置化`
3. 再做 `老虎机与三口味反馈`
4. 最后做 `中后期关卡重调`

原因：
- 先做 P0，才能知道当前关卡是否真的可玩
- 先统一目标，再谈数值调整才有意义
- 先把反馈做清楚，再上更复杂的关卡体验

## 下一步建议

如果按工程推进，下一轮最适合直接落地的是：

### Phase A
- 接入关卡目标进度
- 接入胜利判定
- 接入失败判定
- 在 UI 中显示目标和剩余限制

### Phase B
- 让顾客队列完全读取关卡配置
- 明确顾客与关卡目标的关系
- 补对应回放测试

### Phase C
- 优化老虎机说明与反馈
- 调整安全生成策略
- 回调第21关后的数值曲线
